## Step 2: Mapping

### Opportunity 1: Prose-heavy process encoding

**Scope:** NQ-app
**Domain:** process
**Priority:** normal
**Core question:** How to shift from prose-as-instruction to structural defaults.
**Library — primary:** Stack Builders Beyond AGENTS.md (deterministic named commands, personas); thedecipherist Starter Kit (slash commands + hooks + rules as structural enforcement); MDD (two-zone `.startup.md` priming).
**Library — secondary:** Fowler Context Anchoring; AI Corner persistent file system.
**Petri reference patterns:** The entire `.claude/skills/` directory is the working alternative to prose-as-instruction. Specifically `/new-phase`, `/refine-feature`, `/implement-feature`, `/fix-bug`, `/update-docs`, `/retro` — each is a concrete workflow codification of what NQ currently keeps in CLAUDE.md prose.
**Note:** Strong coverage.

### Opportunity 2: No formal retro mechanism

**Scope:** NQ-app
**Domain:** process
**Priority:** normal
**Core question:** How to capture and compound friction signals across sessions.
**Library — primary:** Fowler Feedback Flywheel (the four signal types — context gaps, instruction patterns, workflow sequences, failure root causes; brief post-session reflection; periodic artifact reviews).
**Library — secondary:** *(none)*
**Petri reference patterns:** `/retro` skill. Concrete multi-step protocol with history search, six friction-signal categories (interrupted actions, clarifications, context slips, human-caught bugs, intensive processes, value signals), and routing rules for proposals (design values → Values.md; communication norms → CLAUDE.md; workflow guardrails → the relevant skill).
**Note:** Clean single-source library match plus a working Petri implementation.

### Opportunity 3: Documentation fan-out with no named coordinator

**Scope:** NQ-app
**Domain:** process
**Priority:** normal
**Core question:** How to manage multi-file doc updates per phase completion without drift.
**Library — primary:** thedecipherist Starter Kit (hooks); Claudefa.st Sub-Agent Best Practices (a docs-coordinating sub-agent); PopularAiTools (Operator / central orchestrator pattern).
**Library — secondary:** Stack Builders deterministic commands like `$commit`; MDD documentation discipline.
**Petri reference patterns:** `/update-docs` skill. Explicit audience-discipline rules per file (README for players, CLAUDE.md for AI context, architecture.md for patterns, game-mechanics.md for player behavior, etc.). Enforces `[TEST] → [DOCS] → [RETRO]` as a unit at every testable milestone.
**Note:** Strong coverage across library and Petri.

### Opportunity 4: Minimal persistent per-phase state *(lower priority)*

**Scope:** NQ-app
**Domain:** process
**Priority:** lower
**Core question:** What could carry better across sessions beyond design docs + CLAUDE.md.
**Library — primary:** Fowler Context Anchoring (lightweight living feature doc); MDD startup file pattern.
**Library — secondary:** *(none)*
**Petri reference patterns:** Phase design doc (`docs/<phase>-design.md`) with DD-N numbering convention. DDs are globally numbered, cross-referenced from step specs; even when `step-spec.md` is replaced, the DDs persist. This is Petri's answer to "what carries across sessions."

### Opportunity 6: `db.rs` as a 7,557-line monolith

**Scope:** NQ-app
**Domain:** structure
**Priority:** normal
**Core question:** When and how to split a large domain file.
**Library — primary:** Fowler LLMs and Abstractions (evolutionary design); Eric Evans (bounded contexts, strategic design); Schleicher (ubiquitous language at module boundaries).
**Library — secondary:** *(none)*
**Petri reference patterns:** `docs/triggered-enhancements.md` contains an explicit "architecture.md split" entry with trigger conditions (>~1000 lines, section-rewrite need, drift-unnoticed staleness). Pattern: document the planned split in advance with trigger conditions, rather than splitting preemptively or waiting for pain.
**Note:** **Gap.** Library has no direct "how to plan a monolith split" piece. Petri's triggered-enhancement pattern is the closest applicable process pattern. DDD framing provides general direction.

**Scope:** NQ-app
**Domain:** structure
**Priority:** normal
**Core question:** How to work through the three flagged tech-debt items.
**Library — primary:** Fowler LLMs and Abstractions (LLMs strong at accidental complexity — mechanical refactors fit this well).
**Library — secondary:** Claudefa.st Sub-Agent Best Practices (dispatch a tech-debt sub-agent).

**Note:** Coverage is adequate — the tech-debt items are concrete enough that framing the work is easy. Library informs approach; Petri informs staging.

### Opportunity 8: Five completion paths coupling

**Scope:** NQ-app
**Domain:** structure
**Priority:** normal
**Core question:** How to ensure shared downstream effects fire consistently across all paths.
**Library — primary:** Schleicher (ubiquitous language — "what does completion mean in this system"); Fowler LLMs and Abstractions (extract shared abstraction).
**Library — secondary:** Hamel Evals FAQ (behavioral-consistency test patterns — adjacent but useful for regression coverage).
**Petri reference patterns:** Values.md "Follow the Existing Shape" plus the sibling-flow-check step in `/fix-bug` (Step 3: "Identify flows that share the same structure as the affected code. Check each for the same issue.") Plus the shared-helper pattern: `RunVesselProcurement`, `RunWaterFill` in `picking.go` extract shared multi-caller logic behind a status-enum return.
**Note:** **Gap** in library for direct multi-path coupling patterns. Petri has the stronger process answer (sibling-flow check) and the stronger structural answer (shared-helper extraction).

Four opportunities where library coverage is thin or absent:

- **#6 `db.rs` monolith split** — no direct library piece; Petri has a staging pattern (triggered-enhancements with conditions) but the substantive "when to split" judgment isn't library-sourced.
- **#8 Multi-path completion coupling** — no direct library pattern; Petri has both the process (sibling-flow check) and structure (shared helpers) answers.
- **#9 Vanilla frontend framework decision** — library not the right source at all.
- **#11 Cross-project skill coordination** — closest library fit is multi-agent coordination within a single project, which is adjacent but not exact.

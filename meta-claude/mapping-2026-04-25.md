## Step 1: Opportunity inventory

### Goal 1 — Skills that encode and chain the analysis flow

1. **Encode each analysis phase as an invokable skill.** *Process. Normal.* Skills for Phase 1a (methodology), Phase 1b (structure or artifact-map), Phase 3.1 (opportunity inventory), Phase 3.2 (mapping), Phase 3.3 (proposals). Each skill consumes named inputs and produces a named artifact.

2. **Define explicit handoffs between phase skills.** *Process. Normal.* Output of one skill is the input contract of the next. Without explicit handoffs, the chain becomes prose-coupled and brittle.

3. **Make the artifact-map alternative a first-class option in the structure phase.** *Process. Normal.* Today this analysis is the only example of substituting an artifact map for Phase 1b; the substitution should be selectable, not improvised. *(Combined with #15 — same fix, different surface.)*

### Goal 2 — Decision learning

4. **Capture per-article decision records with reasoning.** *Process. Normal.* Each intake decision (keep / drop / promote / replace) saved as a structured record with PM reasoning, not just final state. Today decisions are baked into review reports and lost as state.

5. **Build PM-preference memory from accumulated decisions.** *Process. Normal.* Auto-memory entries (feedback type) capture recurring patterns: what kinds of pieces get rejected, what framing language gets pushed back on, what credibility signals matter most. Sharpens future scan recommendations.

6. **Make scan recommendations confidence-rated using accumulated decisions.** *Process. Lower.* Once decision history exists, surface a confidence score per recommendation. Lower because requires #4 and #5 to be in place first.

### Goal 3 — Tighter intake flow

7. **Replace bulk review reports with interactive per-article review.** *Process. Normal.* Scan surfaces candidates one at a time with summary + recommendation; PM accepts/rejects/defers in conversation; final report is the byproduct, not the primary artifact.

8. **Retire the intake doc as a staging area.** *Process. Normal.* PM has confirmed it's not used. Decisions go directly to catalog (promote) or to a decision-record file (drop, with reasoning). The two-bucket intake pattern goes away.

9. **Make intake-review snapshots a session log, not a deliverable.** *Process. Lower.* After #7, reviews become append-only logs of what was reviewed when, with full reasoning living in the decision records (#4).

### Goal 4 — Cross-project automation

10. **Push proposals from meta-claude into target-project intake.** *Cross-project. Normal.* Phase 3.3 output today lives in meta-claude's directory. A handoff mechanism makes proposals actionable in the project, not just visible in meta-claude.

11. **Track proposal uptake.** *Cross-project. Normal.* Which proposals were acted on, deferred, rejected, with reasoning. Closes the proposals → actual change loop. Feeds back into refining the process the same way intake decisions refine library curation.

12. **Trigger analysis passes on project-change signals.** *Cross-project. Lower.* New phase docs in Petri, large skill changes in NQ, or accumulated retros could signal an analysis pass is due. Lower because requires #10 and #11 first.

13. **Periodic re-analysis cadence.** *Cross-project. Normal.* A scheduled remote agent that re-runs Phase 1a on a project quarterly, surfaces deltas from prior analysis, and flags whether a new full pass is warranted.

14. **Cross-project pattern surfacing.** *Cross-project. Lower.* When a proposal type appears in two project analyses, surface as a candidate for a generalized pattern in the library or `analysis-process.md`. Today hand-spotted; could be automated as the corpus grows.

### From prior analysis (not in stated goals)

15. **Codify the artifact-map alternative in `analysis-process.md`.** *Process. Normal.* Combined with #3.

16. **Cross-pass synthesis document.** *Process. Lower.* As analyses accumulate, a doc summarizing recurring themes across passes. Adjacent to #14 but project-local.

17. **Structural enforcement of "surface reasoning in chat" rule.** *Process. Lower.* Today the rule is prose. Hooks could check that proposal docs don't introduce framings not present in the conversation.

## Step 2: Mapping

### Opportunity 1: Encode each analysis phase as an invokable skill

**Core question:** How should the methodology be encoded as runnable skills rather than prose?

- **Library primary:** Trensee five-layer architecture (skills are for procedures); Stack Builders Beyond AGENTS.md (deterministic named commands per phase); PopularAiTools Workflow Patterns (workflow taxonomy).
- **Library secondary:** Subagents reference (when phases benefit from isolated context).
- **Sibling patterns:** Petri's skill graph — `/new-phase` → `/refine-feature` → `/implement-feature` → `/retro` is the most concrete model. NQ's 8-step cycle is the same pattern at finer granularity.
- **Note:** Strong sibling pattern; library entries are conceptual backing.

### Opportunity 2: Define explicit handoffs between phase skills

**Core question:** How do skills cleanly hand off state to the next skill?

- **Library primary:** Fowler Context Anchoring (externalize decisions into persistent docs that survive context loss); MDD (two-prompt architecture, separating audit from implementation, prevents context-compaction failures).
- **Library secondary:** get-shit-done (atomic task plans, structured markdown loaded strategically).
- **Sibling patterns:** Petri's anchor-story propagation — phase design → step spec → anchor test, the same intent statement is the handoff at each stage. Petri's DD-N numbered decision pattern (cross-referenced decisions surviving context).
- **Note:** Petri patterns are mature enough to lift directly.

### Opportunity 3 + 15: Artifact-map alternative as first-class option

**Core question:** How should the structure phase branch for non-software projects?

- **Library primary:** None directly — methodology-internal refinement.
- **Library secondary:** Addy Good Spec (modular task division — variant phases for variant project types, light fit).
- **Sibling patterns:** None — Petri and NQ are both software.
- **Note:** **Gap.** This is process-document scope, not library scope. Resolves with an edit to `analysis-process.md` and a branch in the structure-phase skill. No library lift available.

### Opportunity 4: Capture per-article decision records with reasoning

**Core question:** What's the shape of a structured decision-record artifact that feeds learning?

- **Library primary:** Fowler Feedback Flywheel — names four signal types (context gaps, instruction patterns, workflow sequences, failure root causes) and feeds them into shared artifacts. Direct match.
- **Library secondary:** Addy Agent Harness Engineering — Ratchet Principle (each failure becomes a permanent design constraint encoded into rules). Treating each "drop" decision as a constraint to encode.
- **Sibling patterns:** Petri's `/retro` skill is the practical template — assesses friction across six categories, proposes routed changes, implements on approval. The categorized friction-signal model maps directly to per-article decision categories.

### Opportunity 5: PM-preference memory from accumulated decisions

**Core question:** What belongs in auto-memory vs. dated decision files?

- **Library primary:** Memory (auto-memory) — feedback type is purpose-built for "drop doing X / keep doing X" patterns. The four categories give clear destinations.
- **Library secondary:** Claudefa.st Auto Dream (consolidation when memory accumulates).
- **Sibling patterns:** Petri's `memory/last-retro.txt` is minimal-but-functional. Petri's Values.md ("derived from observed behavior, not prescribed") is the conceptual analogue at file scope.
- **Note:** Strong fit. The library has direct platform-feature coverage.

### Opportunity 6: Confidence-rated scan recommendations

**Core question:** Once decision history exists, how should it influence future recommendations?

- **Library primary:** Hamel Evals FAQ (LLM-as-judge methodology — the scan is an LLM judge; decision history is the validation signal).
- **Library secondary:** Hamel Revenge of the Data Scientist — important warning: confidence-rating without rater validation is one of the named failure modes ("LLM evaluators that haven't been validated themselves").
- **Sibling patterns:** None.
- **Note:** Do not ship confidence ratings until the rater is validated against accumulated decisions. The Hamel Revenge piece is the load-bearing source here.

### Opportunity 7: Replace bulk review reports with interactive per-article review

**Core question:** When should AI batch results vs. surface them one-at-a-time for interactive decisions?

- **Library primary:** Addy Code Agent Orchestra — conductor (real-time) vs. orchestrator (autonomous) framing. Intake review is a deliberate downshift from orchestrator to conductor.
- **Library secondary:** AI Corner power user guide (Socratic prompting); Stack Builders (interactive named commands).
- **Sibling patterns:** Petri's discussion-first pattern — `/refine-feature` Step 2a halt; `/fix-bug` evidence-then-agree halt. Both are explicit "halt for PM confirmation" patterns. Lift directly.

### Opportunity 8: Retire intake doc as staging area

**Core question:** Is the two-bucket staging actually load-bearing?

- **Library secondary:** Addy "Stop Using /init for AGENTS.md" — adapted heuristic: "is this artifact actually used? If no, delete it."
- **Sibling patterns:** None.
- **Note:** Mostly common-sense pruning. Light library coverage; load-bearing input is the PM's "we don't use it" confirmation.

### Opportunity 9: Intake-review snapshots as session log not deliverable

**Core question:** What role do dated reviews play after interactive review takes over?

- **Library secondary:** Fowler Context Anchoring (persistent decision record as audit trail).
- **Sibling patterns:** Petri Values.md is "derived from observed behavior" — analogous shape: behavior happens, log captures it post-hoc.
- **Note:** Light coverage. Re-purposing decision more than a designed change.

### Opportunity 10: Push proposals into target-project intake

**Core question:** How do proposals become actionable artifacts in the target project?

- **Library primary:** Fowler Context Anchoring (externalize decisions into persistent documents in the project that owns the work).
- **Library secondary:** Routines (cloud trigger could push proposals as a PR or file commit); Claudefa.st Hooks Guide (hooks can react to proposal-arrival events).
- **Sibling patterns:** Petri's `triggered-enhancements.md` is the strongest direct template — deferred work with explicit trigger conditions ("when ANY is met"). Meta-claude proposals could land in a project's analogous file with trigger conditions, replacing "this is on the roadmap" vagueness.

### Opportunity 11: Track proposal uptake

**Core question:** How does meta-claude know whether a proposal was acted on?

- **Library primary:** Fowler Feedback Flywheel (capture signals → feed back into shared artifacts).
- **Library secondary:** Hamel Revenge ("look at the data" — treat proposal uptake as the data to look at).
- **Sibling patterns:** Petri's `/retro` + `triggered-enhancements.md` lifecycle — entries created, trigger fires, entries resolve or expire. Same shape needed for proposal tracking.

### Opportunity 12: Trigger analysis on project-change signals

**Core question:** What signals warrant automated re-analysis?

- **Library primary:** Routines — GitHub events trigger is purpose-built (e.g., "new phase doc committed in Petri" → trigger meta-claude scan); Monitor Tool (observe system state).
- **Sibling patterns:** None — neither Petri nor NQ have automated triggers themselves.
- **Note:** Strong primary. Cleanest cross-project automation primitive in the library.

### Opportunity 13: Periodic re-analysis cadence

**Core question:** What cadence and shape should automated re-analysis take?

- **Library primary:** Routines (scheduled trigger).
- **Library secondary:** Claudefa.st Auto Dream (periodic-consolidation cadence-design template).
- **Sibling patterns:** The library-intake-scan skill itself — meta-claude already has one Friday-noon scheduled remote agent. The pattern is proven; this is replicating it for analysis passes.

### Opportunity 14: Cross-project pattern surfacing

**Core question:** How does meta-claude detect when a pattern recurs across multiple project analyses?

- **Library primary:** Fowler Feedback Flywheel (recurring-pattern capture is its central practice).
- **Library secondary:** Schleicher Spec-Driven (ubiquitous language — patterns get named once, recur with the same vocabulary).
- **Sibling patterns:** Petri's `/retro` routing rules — recurring patterns get routed to the appropriate destination (Values.md, CLAUDE.md, skill, architecture.md). Same shape applied across projects rather than within one.

### Opportunity 16: Cross-pass synthesis document

**Core question:** When and how to produce a meta-document of recurring themes across passes?

- **Library primary:** Fowler Feedback Flywheel.
- **Sibling patterns:** None.
- **Note:** Light coverage. May emerge as a need rather than a designed artifact. Adjacent to #14.

### Opportunity 17: Structural enforcement of "surface in chat" rule

**Core question:** Should prose discipline move to structural enforcement here?

- **Library primary:** Fowler Harness Engineering (feedforward/feedback — structural enforcement before failure); Claudefa.st Hooks Guide (PreToolUse blocking pattern).
- **Library secondary:** Addy Agent Harness Engineering (Ratchet Principle).
- **Sibling patterns:** Petri does not use hooks anywhere — same gap. Worth noting that the structural-enforcement gap is shared across both projects.
- **Note:** Lower priority per inventory. Library coverage strong if it gets re-prioritized.

## Cross-cutting themes (strategy checkpoint)

Per `analysis-process.md`, themes spanning multiple opportunities need a strategy discussion before drafting individual proposals. Three themes surfaced:

**Theme A — Skills-as-runnable-workflows.** Opportunities 1, 2, 3, 7, 10, 13. The central engine of all four PM-stated goals. Library: Trensee, Stack Builders, PopularAiTools, get-shit-done. Sibling: Petri's full skill graph (most concrete).

**Theme B — Feedback loops that close.** Opportunities 4, 5, 6, 11, 14. Turning observation into structural improvement. Library: Fowler Feedback Flywheel is the anchor entry. Sibling: Petri's `/retro` (friction signals → routed changes). The loop is currently broken at multiple points (article decisions lost, proposal uptake invisible, cross-project patterns hand-spotted) — worth treating as one designed system rather than separate proposals.

**Theme C — Cross-project automation infrastructure.** Opportunities 10, 12, 13. Genuinely new territory; no sibling pattern exists. Library: Routines is the load-bearing primitive. Most ambitious theme; some of these may need scoping down to manual-but-structured before reaching for automation.

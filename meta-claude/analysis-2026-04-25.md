# Meta-Claude: Working Methodology and Artifact Map

*Analysis performed 2026-04-25*

Meta-claude is the working directory in which Petri and NQ are analyzed and the source library of AI-coding practices is curated. Unlike Petri or NQ, it is not a software product — it produces documents and proposals, and its "running system" is a methodology applied by a PM and an AI together. This analysis follows the standard methodology shape from `analysis-process.md`. The structure-analysis half (Phase 1b) is replaced with a slimmer artifact map, because meta-claude has no stack, no architectural spine, and no code layout to describe — its "structure" is a set of markdown documents and how they relate. The sibling-comparison section is omitted (meta-claude has no sibling).

## Part 1 — Methodology

### What meta-claude is, in one paragraph

Meta-claude is a **methodology workspace** — a directory containing (1) a documented process for analyzing other projects and producing improvement proposals, (2) a curated library of AI-collaboration practices that feeds those analyses, (3) a single skill that scans for new candidates for the library, and (4) the dated outputs of past analysis passes. It runs a three-phase cycle (project analysis → library curation → cross-reference and proposals) with explicit PM checkpoints between phases. The work is recursive: the methodology document itself is updated as lessons emerge, and the library it draws from is curated by the same PM running the analyses.

### Core principles (the stable layer)

Four commitments recur across the process and the artifacts:

1. **Phase the work, gate on alignment.** The most important judgment is at the framing stage — what counts as an opportunity, what library entries actually apply — not at the proposal-writing stage. Phases 1, 2, and 3 are explicitly separated, and Phase 3 itself splits into three sub-steps (inventory → mapping → proposal), each with a checkpoint. The pattern: surface the choice in chat, let the PM redirect, then produce the artifact.

2. **Two parallel tracks, one feeding the other.** Per-project analysis and library curation run in parallel rather than strictly sequentially. The library exists to inform analyses; analyses surface gaps that drive library growth. Neither dominates.

3. **Source attribution is structured.** The catalog uses a fixed two-field schema per entry (Topics + Reach for it when). Proposals use a fixed schema (addresses, change, value, sources, effort, fit, trade-off, recommendation, tier). Schemas mean the artifacts can be compared, scanned, and eventually parsed.

4. **Genericize topics, version the library.** Analysis docs refer to topics ("context anchoring practices"), not article titles ("Fowler's Context Anchoring piece"). The library is mutable; analyses should age more slowly. This is a discipline applied at write time.

### The workflow backbone

The three-phase cycle:

| Phase | Output | Checkpoint |
|---|---|---|
| **1a** Methodology analysis | `<project>-methodology-YYYY-MM-DD.md` | PM reads, flags contradictions/missing context |
| **1b** Structure analysis | `<project>-structure-YYYY-MM-DD.md` | Same |
| **2** Library/catalog curation | Updates to catalog and/or intake | Runs in parallel, no formal gate |
| **3.1** Opportunity inventory | Numbered list, scope-grouped (chat) | PM confirms inventory before mapping |
| **3.2** Opportunity → source mapping | Per-opportunity mapping (chat or doc) | PM checks for force-fits and missed entries |
| **3.3** Proposal document | `<project>-proposals-YYYY-MM-DD.md` | Optional review checkpoint |

It is a graph, not a strict pipeline. Library curation feeds back into Phases 1 and 3 throughout. Proposal drafting can surface a gap that prompts a library scan. The process document captures this as "two parallel tracks" rather than imposing a sequential flow.

### Distinctive methodological moves

Worth naming, because these are the parts that diverge from generic project-review or audit practices:

1. **Two-bucket intake with explicit lifecycle.** The intake file (`claude-code-source-library.md`) has two named sections — `To Be Evaluated` (durable-if-good) and `Latest Updates` (time-bound / release-adjacent). Each entry's eventual destiny — promote to catalog, drop, or expire — is signaled by which bucket it lives in. The catalog (`claude-code-source-library-catalog.md`) is the promoted set with structured annotations. Movement between buckets is an explicit curation act, not silent decay.

2. **Tier-1/2/3 source classification for scanning.** The library-intake-scan skill names sources by tier: Tier 1 (actively checked, 8 sources), Tier 2 (priority-if-surfaces, 6 library-known domains), Tier 3 (broader search). A Tier 2 hit carries elevated credibility presumption because the library already trusts that domain. This is institutional knowledge encoded into the scan procedure.

3. **Sibling-project patterns as a first-class source.** Phase 3 mapping looks at three source types in parallel: catalog primary, catalog secondary, sibling-project patterns. A pattern from Petri's `/retro` skill can drive an NQ proposal as legitimately as a library article. This was a deliberate finding from the NQ pass — sibling patterns were often stronger sources than library entries.

4. **`value` vs. `fit` distinction in proposals.** A proposal whose `value` reduces to "the library names this pattern" is speculative. The schema separates `value` (specific pain or value, grounded in project reality) from `fit` (why this *shape* of change suits the project's existing practice). The discipline is to make sure both can be articulated independently before writing the proposal up.

5. **Cross-cutting strategy checkpoints.** When mapping surfaces a theme that spans multiple opportunities ("memory system is unleveraged," "hooks are not used"), the process pauses for a strategy discussion before drafting individual proposals. Without this, theme-driven proposals become speculative pattern-application rather than grounded change. The skill audit (Phase 1a §5b) is a common source of cross-cutting themes.

6. **Genericized references, dated snapshots.** Analysis documents are dated in their filenames and reference topics rather than article titles. The intent is that an NQ analysis from 2026-04-18 should remain readable in 2027 even if the catalog has churned in the meantime. The catalog itself is undated and mutable.

7. **Claude is opinionated where the PM has veto, and surfaces the reasoning in chat.** Three decision points have explicit "Claude proposes, PM confirms in chat" handling: gap-handling choice (intake / skill / reference doc / nothing), sibling-pattern transfer, and library-scope edge cases during intake. The reasoning must surface in chat at the point of decision, not buried in a multi-section document. Burying an opinion is the failure mode this rule guards against.

8. **Intake-scan as a separate, periodic skill.** Library curation has its own automation (library-intake-scan) that runs on a Friday-noon cadence via a scheduled remote agent, independent of any analysis pass. This decouples library growth from project work — the library can keep up with platform features and community writing without waiting for the next analysis cycle.

### Skill audit

Meta-claude has one skill (`library-intake-scan`) and no custom agents. The skill is a relatively recent addition — most of meta-claude's work has been hand-driven analysis, and the skill formalizes only the periodic library-scan piece.

#### `library-intake-scan`
Periodic web scan across Tier-1 actively-checked sources, Tier-3 broader-search queries, GitHub trending, HN, and Reddit. De-duplicates against catalog + intake + recent reviews. Evaluates each candidate against five criteria (source credibility, durability, coverage complement, practitioner depth, community consensus). Outputs a dated review report to `intake-reviews/`. Designed to run via a scheduled remote agent on Friday noons; can also be invoked ad hoc.

- **Overlap with platform features:** None significant. Memory is not used to retain notes across runs, but the dated review files act as durable cross-run state and that may be the right level for this purpose.
- **Outdated patterns:** None observed; the skill was written recently against the current platform.
- **Unleveraged capabilities:** No hooks. The "don't re-surface what's already in catalog/intake/recent-reviews" discipline is enforced by reading those files at the start of each run, not by structural validation. This appears appropriate for the skill's scope.

The single-skill scope means meta-claude's methodology is overwhelmingly prose-driven (process documents, schemas, checkpoint discipline) rather than tool-mediated. Whether more of the analysis process should be automated is an open question — the analysis-process document lists "Automation candidates" at the end (opportunity extraction, mapping first-draft, tier assignment, proposal scaffolding), but none have been built yet.

### What this methodology optimizes for

- **Source-grounded change.** The library exists so that proposals are not just pattern-matching to whatever Claude has read recently. Every proposal cites both library entries and sibling patterns. The discipline is "no proposal without grounded value," and the schemas are designed to make ungrounded proposals visible.
- **Catching the framing error early.** Two checkpoints sit before proposal drafting (after Phase 1, after Phase 3.2 mapping). Misalignment caught in the inventory or mapping stages costs minutes; misalignment caught in proposals costs an entire draft. The phase split is paying down that risk.
- **Methodology that improves itself.** The lessons-learned section at the bottom of analysis-process.md (currently five entries from the NQ pass) is the explicit feedback loop. The process document is the artifact most likely to be edited by future analysis passes — and it has been, between NQ and Petri.
- **Two continuous signals for adoption.** The library is scoped to (a) new Claude Code/Anthropic platform features and (b) emerging community-consensus patterns. The PM uses both as input to project decisions; the library and proposal pipeline is the mechanism that turns those signals into project-specific change.

### Is the process serving its goals?

This is honest about its limits: meta-claude has produced one complete cycle (NQ, 2026-04-18) and is mid-cycle on a second (Petri, methodology + structure complete, proposals partially drafted). Two data points is too few to draw strong conclusions, and runtime observation of how the process feels during use is not available from this document analysis.

**What looks well-served:**
- **The process produced a usable artifact set.** The NQ pass output (methodology, structure, mapping, proposals) has the schema discipline and source attribution the process is designed to produce. Petri's documents follow the same shape, and the one-week gap between the two passes was enough to fold lessons (codified as the five "Lessons learned" entries) into the second run.
- **Library curation is keeping pace with platform change.** Two intake reviews in six days have evaluated 20 articles and flagged 11 platform updates. The catalog has grown from a smaller initial seed to ~30 entries. The intake-scan skill has been used in production rather than gathering dust.
- **The phase-gating discipline appears to hold.** The NQ proposals document references the Phase 1 documents and the mapping by name; the work was not collapsed into a single pass. The "lessons learned" section explicitly documents one case (the "skill" framing was over-applied in NQ proposals) where a checkpoint caught a real misalignment, which is what the checkpoints are for.
- **Sibling-project sourcing emerged as expected.** Petri's `/retro` and `/update-docs` skills drove concrete NQ proposals; this is documented in the lessons section as something to expect on subsequent passes. The asymmetry — library primary for first project, sibling patterns heavy for subsequent ones — is named and now baked into the process.

**Pressure points and opportunities:**

- **Sample size is two.** Whether the process generalizes to project types unlike Petri or NQ (e.g., a corporate codebase, a frontend-heavy product, a system without long-running PM ownership) is unknown. The current schemas are written generically but have only been exercised on two narrative-heavy, PM-driven, single-developer-style projects.
- **No sibling for meta-claude itself.** This analysis exposes a wrinkle: meta-claude doesn't have the cross-project sourcing it relies on for other projects. Improvements to the process come from the lessons-learned section alone — a self-feedback loop without external validation. This is fine while the process is novel, but worth flagging if meta-claude itself ever gets analyzed against a sibling methodology workspace.
- **The "structure analysis" half is awkwardly fitted to non-software projects.** Phase 1b's template (stack, architectural spine, code patterns) presupposes a software product. Adapting it for meta-claude required replacing it with an artifact map. If future analyses extend to other non-software projects (a methodology, a library, a knowledge base), the process document should probably name an "artifact-map alternative" explicitly rather than leaving each pass to figure out its own adaptation.
- **Most of the methodology is prose.** Only one piece of the cycle (library-intake scan) has been encoded as a skill. The "Automation candidates" list at the bottom of analysis-process.md names plausible next targets (opportunity extraction, mapping first-draft, proposal scaffolding) but none are built. Whether prose discipline is sufficient or whether parts should move to skills is an open question — likely answerable only after several more passes have produced enough repetition to know what's worth automating.
- **The library-intake-scan / analysis-pass coupling is implicit.** The two pieces don't formally connect: scan reports go to `intake-reviews/`, analysis passes consume the catalog. There's no explicit handoff that says "before starting a new analysis, check whether unread intake reviews have surfaced relevant entries." It works because the same PM runs both, but a more explicit trigger could be useful as the cadence grows.
- **Checkpoint surfacing depends on Claude's discipline.** The "surface reasoning in chat, not buried in docs" rule is a prose instruction, not a structural check. Hooks or a doc-scan validator could enforce that proposal documents don't introduce new framing that hasn't been raised in chat. This is the kind of structural-enforcement-vs-prose-discipline pressure point the catalog itself names.

**What can't be determined from this analysis:**
- Whether the schema overhead feels worth it to the PM during real use, or whether it's quietly creating a "do I really want to start an analysis pass" friction.
- Whether the proposals document is being acted on (changes made to NQ or Petri) or filed and forgotten — the analysis pass produces an artifact, but its uptake is invisible from inside meta-claude.
- Whether the library is large enough to be useful or already too large to scan effectively. ~30 catalog entries is at the boundary where a quick-lookup section becomes load-bearing; whether the existing topic index is sufficient is not visible without watching it in use.
- Whether the cadence is right — every six days for intake scans, project-by-project for analyses. No baseline yet for what cadence sustains the work without overload.

**Overall read.** The methodology is well-designed for the work it's been used on, has demonstrated self-improvement (lessons folded between passes), and produces durable, source-attributed artifacts. The largest open questions are about generalization (sample size of two), automation pacing (one skill out of several plausible candidates), and uptake (proposals → actual project change). None of these are urgent; they sharpen as more passes are run.

### Using this as a lens for the library

This section is recursive in a way it isn't for Petri or NQ. Meta-claude *is* the lens for the library — its purpose is to consume library entries and produce project-specific change. So "what library articles are relevant to meta-claude" reduces to "what articles improve the methodology of analyzing AI-coding projects."

That's a narrower set than the library overall. Articles directly relevant to meta-claude's process improvement:

1. **Spec / design discipline** — the analysis-process document and the per-proposal schema are essentially specs. Articles on spec quality, ambiguity resolution, and design-first collaboration sharpen the schemas.
2. **Feedback loops and retros** — the "lessons learned" section is meta-claude's retro mechanism. Articles on how retros become structural improvements (vs. just notes) apply.
3. **Context engineering** — the analysis docs are context-priming artifacts for future passes. What goes in CLAUDE.md vs. analysis-process.md vs. per-pass docs is a context-architecture question.
4. **Evaluation methodology** — the five inclusion criteria for library entries (credibility, durability, coverage complement, practitioner depth, community consensus) are an eval rubric. Articles on eval validation, judge calibration, and rubric drift apply.
5. **Workflow patterns / orchestration** — the three-phase cycle is a workflow pattern. Articles on phase discipline, gate design, and human-in-the-loop alignment apply.

The library currently has strong coverage of (1), (2), (3), and (5), and lighter coverage of (4) for the rubric-validation angle specifically.

Articles that are likely **low-signal for meta-claude** (even if they're high-signal for projects meta-claude analyzes): runtime LLM orchestration, RAG pipelines, multi-agent coordination at runtime, deployment / scaling / infrastructure. Meta-claude is a methodology workspace, not a runtime system.

## Part 2 — Artifact Map (in lieu of structure analysis)

Meta-claude has no stack, no code, and no architectural spine to describe. What it has is a set of markdown documents and a single skill, organized by purpose and volatility. This map names them and how they relate.

### Document inventory by volatility

| Layer | File / location | Purpose | Volatility |
|---|---|---|---|
| **Methodology spine** | `analysis-process.md` | The process itself, updated as lessons emerge | Low — edited only between analysis passes |
| **Library — promoted** | `claude-code-source-library-catalog.md` | Curated entries with Topics + Reach for it when annotations | Medium — entries added as intake promotes them; rare deletions |
| **Library — staging** | `claude-code-source-library.md` | Two buckets: To Be Evaluated, Latest Updates | High — populated by intake scans, drained by promotion or drop decisions |
| **Per-project analyses** | `<project>-methodology-YYYY-MM-DD.md`, `<project>-structure-YYYY-MM-DD.md` | Phase 1 outputs | Frozen at write — dated snapshots |
| **Per-project mapping** *(optional)* | `<project>-mapping-YYYY-MM-DD.md` | Phase 3.2 output, when complex enough to warrant a doc | Frozen at write |
| **Per-project proposals** | `<project>-proposals-YYYY-MM-DD.md` | Phase 3.3 output, the final deliverable | Frozen at write |
| **Intake review snapshots** | `intake-reviews/intake-review-YYYY-MM-DD.md` | Output of one library-intake-scan run | Frozen at write |
| **Skill** | `.claude/skills/library-intake-scan/SKILL.md` | Periodic library-scan automation | Low — edited as scan procedure refines |
| **Local config** | `.claude/settings.local.json` | Project-local Claude Code settings | Low |

The volatility column matters: dated artifacts are intentionally frozen so they remain useful as historical references, while the catalog and process document are edited in place because their value comes from being current.

### How artifacts feed each other

```
                                  │ analysis-process.md │  ← edited via lessons learned
                                  │  (methodology spine)│

                                             ▼

                            │   (Phases 1, 2, 3)                   │
                            └─────┬──────────────────────┬────────┘
                                  │                      │
                consumes          │                      │ produces
                                  ▼                      ▼
              ┌─────────────────────────────┐    ┌──────────────────────────────┐
              │ catalog (promoted entries)  │    │ <project>-methodology-DATE.md│
              │ + sibling-project patterns  │    │ <project>-structure-DATE.md  │
              └──────────────┬──────────────┘    │ <project>-mapping-DATE.md    │
                             ▲                    │ <project>-proposals-DATE.md │
                             │ promotes from      └──────────────────────────────┘
                             │
              ┌──────────────┴──────────────┐
              │  intake (To Be Evaluated /  │
              │  Latest Updates)            │
              └──────────────┬──────────────┘
                             ▲
                             │ flags candidates
                             │
              ┌──────────────┴──────────────┐
              │  library-intake-scan skill  │  ← runs periodically, produces
              │  → intake-reviews/DATE.md   │     dated review reports
              └─────────────────────────────┘
```

Three feedback edges worth noting:

1. **Lessons → process.** Outputs of an analysis pass can include refinements to `analysis-process.md` (the "Lessons learned" section). This is the only place where the methodology document changes.
2. **Gaps → intake.** A Phase 3.2 mapping that finds no library coverage for an opportunity can result in a new candidate added to intake (per the gap-handling rules in `analysis-process.md`).
3. **Reviews → catalog.** Intake review snapshots flag candidates for catalog promotion; the PM acts on those recommendations manually, moving entries from intake to catalog.

### What meta-claude is NOT

Naming what it isn't clarifies which library practices miss the target — including, recursively, articles in meta-claude's own library:

- **Not a software product.** No stack, no code, no runtime. The library entries about RAG, vector search, deployment, scaling, latency, and observability are about a different class of system entirely.
- **Not an LLM-orchestration system.** No agent fleet, no tool-use loop, no API integration. The "agent" in meta-claude's library means *AI coding assistant for the developer*, not *runtime tool-using LLM*. Library entries about agent teams, multi-agent coordination, and Managed Agents are relevant for projects meta-claude analyzes (Petri, NQ), not for meta-claude itself.
- **Not a knowledge management system.** The library is small, hand-curated, and queried by reading. There's no search index, no embeddings, no automated retrieval. Library entries on contextual retrieval, hybrid search, and RAG are out of scope for meta-claude's own operation.
- **Not a continuous-integration target.** No build, no test suite, no CI pipeline. The "test" of an analysis pass is whether the PM finds the proposals useful and acts on them — there's no automated signal.
- **Not multi-tenant.** One PM, one AI collaborator, one workspace. Library entries on team collaboration, role-based access, and shared state coordination are about projects with multiple human collaborators.
- **Not real-time.** The cadence is days-to-weeks, not seconds. Library entries on streaming responses, latency-sensitive UX, and synchronous coordination are not relevant.

### Implications for which library practices apply to meta-claude

**Highly relevant:** spec/design discipline, retro / feedback-loop patterns, context engineering for AI-assisted work, workflow phasing and gate design, eval rubric design (for the inclusion-criteria rubric), structural-vs-prose enforcement (for whether to encode more checkpoints as hooks).

**Adjacent:** subagent patterns (could be used to parallelize phases of an analysis pass — currently not used), memory patterns (could hold cross-pass insights — currently held in `analysis-process.md`'s lessons section instead).

**Irrelevant or misleading when applied to meta-claude:** RAG and retrieval, multi-agent orchestration, vector databases, deployment patterns, API design, real-time / streaming concerns, runtime observability.

### A note on "agent" as a word

In meta-claude:
- An **AI collaborator** is the Claude Code session driving the analysis. Singular, present tense.
- A **scheduled remote agent** is the cloud-hosted runner that executes the library-intake-scan skill on a Friday-noon cadence. The only autonomous component in the system.
- The library entries that discuss "agents" almost always mean something else — multi-agent orchestration, runtime tool-using LLMs, or developer harnesses for building those — and should be classified as relevant to projects meta-claude analyzes, not to meta-claude itself.

### Where meta-claude might grow

Plausible directions, none committed:

- **Automation candidates already named in `analysis-process.md`** — opportunity extraction from Phase 1 documents, mapping first-draft via topic-matching against catalog Topics fields, proposal-document scaffolding. None built; whether they're worth building depends on how often analysis passes run.
- **An artifact-map template for non-software projects.** This document is the first time Phase 1b has been replaced with an artifact map; if more non-software projects get analyzed, the alternative template should be codified in `analysis-process.md`.
- **Cross-pass synthesis.** Currently each analysis pass produces its own dated artifacts. As the count grows, a meta-document that synthesizes patterns across passes (e.g., "themes recurring in three or more analyses") may become useful.
- **Catalog growth management.** ~30 entries today is browsable. At 60-80 the topic index in the catalog may stop scaling; some kind of structured tagging or filtered views could help. Not urgent yet.

None of these introduce a stack, runtime, or code layer. Meta-claude stays a markdown-and-process workspace.
